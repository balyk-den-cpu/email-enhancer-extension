# Требования и архитектура расширения для улучшения работы с электронной почтой

## 1. Обзор

Данный документ описывает требования и предлагаемую архитектуру интерфейса для браузерного расширения, предназначенного для улучшения функциональности почтовых клиентов (Gmail, Yandex Mail, Outlook, Mail.ru). Расширение будет предоставлять инструменты для голосового ввода, коррекции текста, AI-преобразования текста и настройки пользовательского опыта.

## 2. Функциональные требования

### 2.1. Основные функции

*   **Запись голоса и интерпретация в текст:**
    *   Пользователь должен иметь возможность записать голосовое сообщение.
    *   Записанное сообщение должно быть преобразовано в текст и вставлено в поле ввода письма.
    *   Должна быть индикация записи и возможность ее остановки.

*   **Исправление текста на ошибки:**
    *   Функция должна исправлять орфографические, грамматические и пунктуационные ошибки в выделенном или всем тексте поля ввода.
    *   Должна также улучшать форматирование (отступы, абзацы) для придания письму аккуратного вида.

*   **Преобразование текста с помощью ИИ (Улучшение):**
    *   Пользователь должен иметь возможность выделить часть текста или весь текст в поле ввода.
    *   Выделенный текст должен быть отправлен на AI-обработку для улучшения.
    *   Результат AI-преобразования должен заменять исходный текст.

*   **Выбор стилей преобразования:**
    *   Должен быть доступен список стилей для AI-преобразования:
        *   Дружелюбный
        *   Официальный
        *   Краткий
        *   Убеждающий
        *   Технический

*   **Выбор языка текста:**
    *   Должен быть доступен выбор языка для обработки текста (как для голосового ввода, так и для AI-преобразования/коррекции).
    *   Поддерживаемые языки: Русский, Английский, Испанский, Португальский, Арабский (фусха/MSA), Французский.

### 2.2. Взаимодействие с полем ввода

*   Все функции преобразования текста должны работать с **выделенным текстом** в поле ввода.
*   Если текст не выделен, функции должны применяться ко **всему содержимому** поля ввода.
*   Должна быть возможность регулировать (отменять/повторять) изменения.

### 2.3. Размещение элементов интерфейса

*   **Основные кнопки функций:** Должны быть размещены в виде иконок и/или списков **возле поля ввода сообщения**, рядом с кнопкой 

отправки письма.
*   Должно быть обеспечено аккуратное и неинтрузивное размещение, гармонирующее с существующим интерфейсом почтовых клиентов.

### 2.4. Настройки расширения

*   **Иконка настроек:** Должна быть расположена в верхней части виджета расширения.
*   **Страница настроек:**
    *   **Информация о подписке:** Отображение срока действия подписки.
    *   **Выбор языка интерфейса:** Пользователь должен иметь возможность выбрать язык интерфейса расширения.
    *   Поддерживаемые языки интерфейса: Русский, Английский, Испанский, Португальский, Арабский (фусха/MSA), Французский.
    *   При смене языка интерфейса все элементы расширения должны быть переведены.

## 3. Технические требования и архитектура интерфейса

### 3.1. Общие принципы

*   **Кроссбраузерность:** Расширение должно быть совместимо с основными браузерами (Chrome, Firefox, Edge), хотя первоначальная разработка может быть сфокусирована на Chrome.
*   **Инъекция контента:** Элементы интерфейса расширения должны быть инжектированы в DOM почтовых клиентов.
*   **Изоляция стилей:** Стили расширения должны быть изолированы, чтобы избежать конфликтов с существующими стилями почтовых клиентов.
*   **Масштабируемость:** Архитектура должна предусматривать легкое добавление новых функций и интеграцию с внешними API (например, `https://app.base44.com`).

### 3.2. Структура расширения (предполагаемая)

*   **`manifest.json`:** Файл манифеста расширения, определяющий разрешения, скрипты контента, фоновые скрипты и страницы настроек.
*   **`content.js`:** Скрипты, инжектируемые в страницы почтовых клиентов для манипуляции DOM и добавления элементов интерфейса.
*   **`background.js`:** Фоновый скрипт для обработки логики, не связанной напрямую с DOM, такой как взаимодействие с API и управление состоянием.
*   **`popup.html`/`popup.js`:** (Опционально) Интерфейс для настроек или быстрого доступа к функциям, если это будет удобнее, чем отдельная страница настроек.
*   **`options.html`/`options.js`:** Страница для управления настройками расширения.
*   **`styles/`:** Директория для CSS-файлов, обеспечивающих внешний вид элементов расширения.
*   **`assets/`:** Директория для иконок и других медиа-ресурсов.

### 3.3. Размещение элементов интерфейса (детализация)

*   **Панель инструментов функций:** Предполагается размещение в виде небольшой плавающей или прикрепленной панели рядом с полем ввода сообщения. Это может быть `div` элемент, инжектированный в DOM, с использованием `position: absolute` или `position: fixed` и `z-index` для обеспечения видимости.
*   **Иконки функций:** Каждая функция (запись голоса, исправление, AI-улучшение) будет представлена иконкой. При нажатии на AI-улучшение будет появляться выпадающий список со стилями преобразования.
*   **Выбор языка текста:** Может быть реализован как выпадающий список или отдельная иконка, открывающая модальное окно/поповер с выбором языка.
*   **Иконка настроек:** Будет располагаться в верхней части панели инструментов функций или в отдельном виджете, открывающем страницу `options.html`.

## 4. Интеграция с почтовыми клиентами

*   **Обнаружение поля ввода:** Необходимо разработать механизмы для надежного обнаружения поля ввода сообщения в различных почтовых клиентах (Gmail, Yandex Mail, Outlook, Mail.ru). Это может потребовать использования специфичных для каждого клиента селекторов DOM.
*   **Внедрение UI:** Элементы UI расширения будут внедряться рядом с обнаруженным полем ввода.
*   **Обработка событий:** Расширение должно перехватывать события ввода текста и выделения для применения своих функций.

## 5. Взаимодействие с внешним API

*   Расширение будет взаимодействовать с `https://app.base44.com` через эндпоинты для выполнения операций по исправлению и преобразованию текста.
*   Все запросы к API должны быть асинхронными и обрабатываться в фоновом скрипте для обеспечения отзывчивости интерфейса.

## 6. Дальнейшие шаги

1.  Создание базовой структуры расширения.
2.  Разработка `manifest.json`.
3.  Создание `content.js` для инъекции UI и обнаружения полей ввода.
4.  Разработка UI-компонентов (иконки, выпадающие списки, страница настроек).
5.  Реализация логики взаимодействия с полем ввода.
6.  Интеграция с API `https://app.base44.com`.
7.  Тестирование в различных почтовых клиентах.
